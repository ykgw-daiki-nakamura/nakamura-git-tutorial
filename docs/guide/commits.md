# コミットとコミットメッセージ

`git commit` の操作そのものは [Git の基本](./basics) で扱いました。ここでは**チームで読まれることを前提に、どんな単位で・どう書いてコミットするか**を決めます。

コミットは「動く状態のスナップショット」であると同時に、**後から履歴を読む人へのメッセージ**です。1 人で書いているうちは雑でも困りませんが、レビュー・障害調査・リリースノート作成の場面で、過去のコミットは繰り返し読み返されます。

## コミットの粒度

原則は **1 コミット = 1 つの意味のある変更**です。

- **混ぜない** — 機能追加とリファクタリング、本質的な修正と整形を同じコミットに入れない
- **壊れた状態で刻まない** — ビルドやテストが通らない中途半端な状態でコミットしない
- **戻せる形にする** — 粒度が揃っていれば、問題のあった変更だけを `git revert` で取り消せる

粒度を揃えておくと、`git bisect` でバグの混入コミットを絞り込むときにも効きます。逆に「1 日の作業をまとめて 1 コミット」にすると、どの変更が原因かを切り分けられません。

::: tip 作業しながら粒度を整える
最初から綺麗に刻めなくても構いません。`git add -p` で変更を部分的にステージすれば、1 回の作業からでも意味のある単位に分けてコミットできます。
:::

## メッセージの構造

コミットメッセージは **1 行目に要約、空行をはさんで本文**という構造が基本です。

```text
feat: ログインフォームのバリデーションを追加

メールアドレス形式のチェックを追加し、不正な入力時に
エラーメッセージを表示するようにした。

ブラウザ側だけの検証なので、サーバー側の検証は別途必要。
```

- **1 行目（要約）** — 何をしたかを簡潔に。50 文字程度に収める
- **本文** — 「なぜそうしたか」を書く。**何をしたかは diff を読めば分かるので、diff からは読み取れない背景や判断理由**を残す

本文は省略してもかまいません。要約だけで意図が伝わる変更なら 1 行で十分です。

## Conventional Commits

要約の書き方をさらに機械可読な形に決めたのが [Conventional Commits](https://www.conventionalcommits.org/ja/) です。要約の先頭に**変更の種別（type）** を付けます。

```text
<type>(<scope>): <要約>
```

| 部分 | 必須 | 説明 |
| --- | --- | --- |
| `type` | 必須 | 変更の種別。`feat` `fix` `docs` など |
| `scope` | 任意 | 変更した範囲。`(auth)` `(guide)` のようにモジュール名やディレクトリ名を書く |
| 要約 | 必須 | 何をしたか。命令形・現在形で簡潔に |

実際に書くとこうなります。

```text
feat(auth): パスワードリセット機能を追加
fix(api): 空配列を渡すと 500 が返る不具合を修正
docs(guide): ブランチ命名規則の例を追加
chore(deps): eslint を v9 に更新
```

### 主な type

| type | 使うとき |
| --- | --- |
| `feat` | 機能の追加 |
| `fix` | バグの修正 |
| `docs` | ドキュメントのみの変更 |
| `refactor` | 挙動を変えないコードの整理 |
| `test` | テストの追加・修正 |
| `perf` | パフォーマンスの改善 |
| `build` | ビルドシステム・依存関係の変更 |
| `ci` | CI 設定・スクリプトの変更 |
| `style` | 挙動に関わらない整形（空白・セミコロンなど） |
| `revert` | 過去のコミットの取り消し |
| `chore` | 上記のいずれにも当てはまらない雑務 |

どの type を許可するかはプロジェクトごとに決めます。書式を CI で検証している場合は、その設定ファイルが許可一覧の情報源になります。

### 破壊的変更

後方互換性を壊す変更は、type の直後に `!` を付けるか、本文に `BREAKING CHANGE:` から始まる段落を書きます。

```text
feat(api)!: getUser のレスポンス形式を変更

BREAKING CHANGE: getUser が返す `name` を `firstName` と
`lastName` に分割した。呼び出し側の修正が必要。
```

### 何が嬉しいのか

書式を揃えるだけなら好みの問題ですが、Conventional Commits には**機械が読めることによる実利**があります。

- **履歴が検索できる** — `git log --oneline -E --grep '^fix(\(.+\))?!?:'` で修正だけを追える
- **バージョンが決まる** — `fix` は PATCH、`feat` は MINOR、`!` / `BREAKING CHANGE` は MAJOR に対応する（[リリースとバージョン管理](./release#conventional-commits-と対応している) を参照）
- **リリースノートが自動生成できる** — type ごとに変更点を分類してノートに並べられる
- **PR タイトルにも効く** — Squash Merge では PR タイトルがそのまま `main` のコミットメッセージになる（[プルリクエストとレビュー](./pull-request#pr-タイトルは-conventional-commits-に従う) を参照）

つまり、日々のコミットで種別を書いておくと、**リリース時に「次に上げるべきバージョン」と「リリースノートの草案」が履歴から出てくる**わけです。

## 直前のコミットを修正する

push する前なら、直前のコミットは作り直せます。

```bash
# メッセージだけ書き直す
git commit --amend -m "fix(api): 空配列で 500 が返る不具合を修正"

# 直前のコミットに変更を追加する（ファイル漏れの救済）
git add forgotten-file.ts
git commit --amend --no-edit
```

::: warning push 済みのコミットは書き換えない
`--amend` はコミット ID を作り直します。すでに共有ブランチへ push したコミットに使うと、他の人の履歴と食い違います。push 済みの履歴を直したいときの選択肢は [トラブルシューティング](./troubleshooting) を参照してください。
:::

## まとめ

- 1 コミット = 1 つの意味のある変更。壊れた状態で刻まない
- 1 行目に要約、本文には**なぜ**を書く
- 要約は `type(scope): 要約` の形（Conventional Commits）に揃える
- 種別を書いておくと、バージョン決定とリリースノート生成が自動化できる
