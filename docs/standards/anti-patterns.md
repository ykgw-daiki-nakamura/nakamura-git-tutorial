# 禁止事項

やってはいけない運用を、理由とセットでまとめる。全体像は[概要](./)を参照。

## 禁止事項（アンチパターン）

| 禁止事項 | 理由 |
| --- | --- |
| 環境ブランチ（`develop`, `staging`, `production` 等）の作成 | 環境間ドリフトとマージ経路の組み合わせ爆発を招く。環境は GitHub Environments で表現する |
| 顧客別ブランチ（`customer/*`）・顧客別フォークの作成 | パッチ展開コストが「サポート中バージョン数 × 顧客数」に爆発する。カスタマイズは Tier 1（設定/フラグ）または Tier 2（下流リポジトリ）で実現する（[顧客別カスタマイズ](./customization)を参照） |
| 顧客識別子によるコード分岐（`if customer == A` 等） | コードベース内部に実質的なフォークが増殖する。機能単位のフラグ抽象化または拡張ポイントで表現する |
| コアリポジトリへの顧客固有コードの混入 | 顧客間のコード分離（契約境界）が崩れる。Tier 2 は下流リポジトリに置く |
| `main` から staging / production への直接デプロイ | 出荷バージョンの追跡不能化。デプロイは必ずタグ（GitHub Release）起点とする |
| `release/*` で修正し `main` へマージ（upstream first 違反） | main への取り込み漏れにより次バージョンでバグが再発する。修正は常に `main` を起点とする |
| `main` → `release/*` の merge / rebase | release ラインに未検証の新機能が混入する |
| GA タグ・環境ごと・配布形態ごとの再ビルド | ビルドは完全な再現性を持たず、RC で検証した成果物と出荷成果物の同一性が検証不能になる。GA は再タグ付けによる昇格とする（リリースとデプロイの [GA 昇格規約（再ビルドの禁止）](./release#ga-昇格規約-再ビルドの禁止)） |
| `-rc.N` サフィックスの成果物への焼き込み | GA 昇格時に成果物の書き換えが必要になり、再ビルド禁止と矛盾する。RC / GA の区別はタグと Release メタデータで表現する |
| 公開済みタグ・Release の付け直し | 出荷済みバージョンの改変。修正はパッチバージョンで発行する |
| 手動での GitHub Release 作成 | タグ・Release・成果物の対応ずれの原因となる |
| 本番環境への out-of-band 操作（コンソール / kubectl による直接変更・手動ロールバック） | デプロイ履歴と実環境の乖離を生む。ロールバックも必ずパイプライン経由で行う（[障害対応](./incident)の「ロールバック / ロールフォワード方針」） |
| 長期間マージされない feature ブランチ | マージコンフリクトの肥大化。小さな PR に分け、未完成部分は dark launch / 非公開 API としての先行マージ / branch by abstraction のいずれかで隔離して `main` へ入れる（[導入までの暫定規約](./versioning#導入までの暫定規約)）。feature flag は[現時点では利用を見送り検討中](./versioning#feature-flag-運用) |
| 本番資格情報のリポジトリシークレット格納 | 環境スコープ外からの参照リスク。Environment secrets + OIDC を用いる |
| GitHub Actions をタグ・ブランチで参照（`uses: owner/action@v4`） | タグは可変であり、保守者権限を得た攻撃者が同じタグを別のコミットへ付け替えられる。40 桁のコミット SHA でピン留めする（[依存とサプライチェーン](./supply-chain)） |
| `pull_request_target` で PR head のコードをチェックアウト・実行 | 書き込み権限を持つトークンでフォークの任意コードを走らせることになる。検証ロジックとその設定は必ず base 側から読む |
| PR タイトル・本文・ブランチ名を `run:` へ直接展開 | スクリプトインジェクションの経路となる。`env:` 経由で環境変数として渡す |
| lockfile を伴わない依存追加、および CI でのバージョン範囲の再解決 | 解決済みバージョンと完全性ハッシュが固定されず、ビルドごとに異なる依存が入りうる |
| インストール時スクリプト（`postinstall` 等）を有効にしたままの CI | 悪性パッケージが任意コードを実行する主要経路。`ignore-scripts` で無効化する |
| 脆弱性の例外をパッケージ単位で登録、または再評価期限なしで登録 | 前者は同じパッケージの別の脆弱性まで見逃す。後者は放置され、検査が形骸化する。advisory 単位で理由と期限を必須とする |
| 依存更新ボットの PR の自動マージ | 悪性バージョンをレビューなしで取り込む経路になる。CI の成功に加えて人のレビューを要する |
