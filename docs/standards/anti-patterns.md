# 禁止事項

やってはいけない運用を、理由と代わりの手段をセットでまとめる。全体像は[概要](./)を参照。

分類は 3 つ。**ブランチとマージ** / **リリースと出荷** / **セキュリティとサプライチェーン**。

## ブランチとマージ

| 禁止事項 | 理由 | 代わりに |
| --- | --- | --- |
| 環境ブランチ（`develop`, `staging`, `production` 等）の作成 | 環境間ドリフトと、マージ経路の組み合わせ爆発を招く | 環境は GitHub Environments で表現する |
| 顧客別ブランチ（`customer/*`）・顧客別フォークの作成 | パッチ展開コストが「サポート中バージョン数 × 顧客数」に爆発する | Tier 1（設定/フラグ）または Tier 2（下流リポジトリ）で実現する（[顧客別カスタマイズ](./customization)） |
| 顧客識別子によるコード分岐（`if customer == A` 等） | コードベース内部に実質的なフォークが増殖する | 機能単位のフラグ抽象化、または拡張ポイントで表現する |
| コアリポジトリへの顧客固有コードの混入 | 顧客間のコード分離（契約境界）が崩れる | Tier 2 は下流リポジトリに置く |
| `release/*` で修正し `main` へマージ（upstream first 違反） | `main` への取り込み漏れにより、次バージョンでバグが再発する | 修正は常に `main` を起点とする |
| `main` → `release/*` の merge / rebase | release ラインに未検証の新機能が混入する | cherry-pick で必要な修正だけを取り込む |
| 長期間マージされない feature ブランチ | マージコンフリクトが肥大化する | 小さな PR に分け、未完成部分は dark launch / 非公開 API としての先行マージ / branch by abstraction のいずれかで隔離して `main` へ入れる（[導入までの暫定規約](./versioning#導入までの暫定規約)） |

## リリースと出荷

| 禁止事項 | 理由 | 代わりに |
| --- | --- | --- |
| `main` から staging / production への直接デプロイ | 出荷バージョンが追跡不能になる | デプロイは必ずタグ（GitHub Release）起点とする |
| GA タグ・環境ごと・配布形態ごとの再ビルド | ビルドは完全な再現性を持たないため、RC で検証した成果物と出荷成果物の同一性を検証できなくなる | GA は再タグ付けによる昇格とする（[GA 昇格規約（再ビルドの禁止）](./release#ga-昇格規約-再ビルドの禁止)） |
| `-rc.N` サフィックスの成果物への焼き込み | GA 昇格時に成果物の書き換えが必要になり、再ビルド禁止と矛盾する | RC / GA の区別はタグと Release メタデータで表現する |
| 公開済みタグ・Release の付け直し | 出荷済みバージョンの改変にあたる | 修正はパッチバージョンで発行する |
| 手動での GitHub Release 作成 | タグ・Release・成果物の対応ずれの原因になる | タグ push をトリガーにワークフローで作成する |
| 本番環境への out-of-band 操作（コンソール / kubectl による直接変更・手動ロールバック） | デプロイ履歴と実環境が乖離する | ロールバックも必ずパイプライン経由で行う（[障害対応](./incident)） |

## セキュリティとサプライチェーン

| 禁止事項 | 理由 | 代わりに |
| --- | --- | --- |
| 本番資格情報のリポジトリシークレット格納 | 環境スコープ外から参照されうる | Environment secrets + OIDC を用いる |
| GitHub Actions をタグ・ブランチで参照（`uses: owner/action@v4`） | タグは可変で、保守者権限を得た攻撃者が同じタグを別のコミットへ付け替えられる | 40 桁のコミット SHA でピン留めする（[依存とサプライチェーン](./supply-chain)） |
| `pull_request_target` で PR head のコードをチェックアウト・実行 | 書き込み権限を持つトークンで、フォークの任意コードを走らせることになる | 検証ロジックとその設定は必ず base 側から読む |
| PR タイトル・本文・ブランチ名を `run:` へ直接展開 | スクリプトインジェクションの経路になる | `env:` 経由で環境変数として渡す |
| lockfile を伴わない依存追加、および CI でのバージョン範囲の再解決 | 解決済みバージョンと完全性ハッシュが固定されず、ビルドごとに異なる依存が入りうる | lockfile をコミットし、CI は lockfile 固定でインストールする |
| インストール時スクリプト（`postinstall` 等）を有効にしたままの CI | 悪性パッケージが任意コードを実行する主要経路になる | `ignore-scripts` で無効化する |
| 脆弱性の例外をパッケージ単位で登録、または再評価期限なしで登録 | 前者は同じパッケージの別の脆弱性まで見逃す。後者は放置され、何も検知しない検査になる | advisory 単位で登録し、理由と再評価期限を必須とする |
| 依存更新ボットの PR の自動マージ | 悪性バージョンをレビューなしで取り込む経路になる | CI の成功に加えて人のレビューを要する |

feature flag による隔離・縮退に触れる規定は、[現時点では利用を見送っている](./versioning#feature-flag-運用)ため、導入後に適用する。
