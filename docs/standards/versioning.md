# バージョン運用 🚧

> [!WARNING]
> **本ページは検討中**
>
> 記載内容は検討段階の案であり、現時点の確定規約ではない。合意を経て確定するまでは、運用の根拠として参照しない。

バージョニング・サポートポリシーと、Feature Flag 運用を定める。

## このページの要点

- バージョンは SemVer に従う。SaaS は最新 GA のみ、セルフホストは最新（N）と 1 つ前（N-1）をサポートする。
- セルフホストのアップグレードは MINOR を 1 つずつ順に経由する。飛ばさない。
- feature flag は未採用。未完成機能は、ビルド時に隔離する 3 手段のいずれかで `main` へ入れる。

## バージョニングとサポートポリシー

- バージョン番号は SemVer（`MAJOR.MINOR.PATCH`）に従い、リリース候補は `-rc.N` サフィックスで表す。
- **SaaS 本番は常に最新 GA バージョン（N）のみ**を稼働させる。SaaS のバージョンアップは Release 公開後、計画メンテナンスとして実施する。
- セルフホスト版のサポート対象は **最新（N）と 1 つ前（N-1）の MINOR バージョン**とする。
  - N: 機能修正・不具合修正・セキュリティ修正
  - N-1: 不具合修正・セキュリティ修正のみ
- EOL を迎えた `release/*` ブランチは最終タグを付与のうえアーカイブ（read-only 化）する。GitHub Release は EOL 後も削除しない（過去バージョンの追跡可能性を維持する）。

### セルフホスト版のアップグレードパス

- **アップグレードは MINOR を順に経由する**。`v1.1` から `v1.3` へ直接上げることは認めず、`v1.1` → `v1.2` → `v1.3` と 1 MINOR ずつ適用する。同一 MINOR 内の PATCH は最新へ直接上げてよい。
- 理由は、マイグレーションの expand-contract パターンが**隣接バージョン間の互換性しか保証しない**ことにある。保証されるのは「バージョン N のスキーマ上でバージョン N-1 のアプリケーションが動作する」ことだけである（[障害対応](./incident)の「ロールバック可能条件」）。
- MINOR を飛ばすと、そのバージョンで実施されるはずだった expand とデータ移行を経ないまま、次のバージョンの contract（列削除・リネーム・NOT NULL 化）が適用される。スキーマの前提が崩れる。
- サポート対象（N / N-1）とこの必須要件は独立している。顧客が N-1 に留まっている間に次の MINOR が公開されれば、その顧客の現行バージョンはサポート外になり得る。その場合も 1 MINOR ずつ順に適用する。
- 順次適用の手間を減らすため、インストーラは中間バージョンのマイグレーションを順番に流す一括アップグレード手順を提供してよい。その場合も適用の単位と順序は MINOR ごとのままとする。
- MAJOR をまたぐアップグレードの手順は、当該 MAJOR の Release ノートの固定セクション（「アップグレード時の注意」）で個別に定める。

## Feature Flag 運用

> [!NOTE]
> **feature flag は現時点では利用を見送っている**
>
> 本ページ全体が検討中であることに加え、feature flag そのものの採用も見送っている。**後述の「導入後の Feature Flag 運用」は将来の採用に備えた指針であり、導入するまでは適用しない**（本ページ以外で feature flag に言及する箇所も同様の扱い）。導入までの間をどう埋めるかは、次の「導入までの暫定規約」で扱う。

### 導入までの暫定規約

feature flag が使えない間も、[マージルール](./merge-rules#マージルール)の「long-lived な feature ブランチの禁止」は有効である。作りかけの機能は**未完成のまま `main` へマージし、到達不能な状態で隔離する**。

`release/vX.Y` を切る時点で `main` 上に未完成機能が存在してよいのは、次のいずれかの手段で隔離されている場合に限る。

| 手段 | 内容 | 適する場面 |
| --- | --- | --- |
| dark launch | 実装は `main` に入れるが、UI 導線・ルーティング・メニューを繋がない。利用者から到達不能なため、そのまま出荷しても表に出ない | 画面や導線を伴う新機能。導入コストが最も低く、既定の選択肢 |
| 非公開 API としての先行マージ | エンドポイントや拡張ポイントを未ドキュメント・未登録のまま実装だけ先に入れ、公開は別 PR で行う | API・拡張ポイントの追加。公開の可否をリリース直前まで遅らせられる |
| branch by abstraction | 旧実装と新実装を抽象層の背後に並存させ、呼び出し先の切り替えを最後の 1 PR に閉じ込める | 大規模な置き換え・リファクタリング。抽象層の導入コストを払う価値がある場合 |

- どの手段で隔離したかを PR 本文に記載する。`release/vX.Y` を切るリリース責任者は、未完成機能がすべていずれかの手段で隔離されていることを確認する。
- **いずれもビルド時の隔離であり、実行時に稼働中の機能を切り離すことはできない**。障害発生時の機能単位の縮退は、feature flag を導入するまで「該当機能を無効化する緊急パッチ」で行う（[障害対応](./incident)の「ロールバック / ロールフォワード方針」）。
- 隔離のコストがどの手段でも見合わないほど大きい場合は、機能そのものを小さく分割し直す。long-lived な feature ブランチへ退避してはならない（[禁止事項](./anti-patterns)）。

feature flag を導入した際は、dark launch と非公開 API 先行マージはフラグ配下での開発に置き換わる。branch by abstraction は実装技法としてフラグと併用できるため、その後も残す。

### 導入後の Feature Flag 運用

次期バージョン開発中の機能差はブランチではなく feature flag で吸収する。

- 開発中の機能はフラグ配下に置き、`main` へのマージをブロックしない（`dev` 環境ではフラグ ON で検証できる）。
- `release/vX.Y` を切る時点で、各フラグの GA 可否（既定値 ON / OFF / コード削除）を判定する。
- テナント別・顧客別のフラグ制御は設定として管理し、ビルド成果物は SaaS / セルフホストで共通とする。
- 役目を終えたフラグは 2 リリース以内に削除する（フラグ負債の防止）。
