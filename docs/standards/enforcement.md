# 規約の担保状況

本セクションの各規約が、何によって守られるかを一覧にする。

> [!NOTE]
> **一覧に書くのは「この規約が想定する担保手段」である**
>
> 本セクションは架空の製品の開発規約であり、このチュートリアルのリポジトリに現在入っている設定を説明したものではない。以下の表も、規約が前提とする設定・ワークフローを並べたものである。自分のリポジトリへ持ち込むときは、対応する設定が実在するかを別途確認すること。

## このページの要点

- 規約は担保のされ方で 3 つに分かれる。**強制**（違反すると止まる）・**検知のみ**（止まらないが記録される）・**運用**（人が守る）。
- 項目数では**運用**が最も多い。設定を入れれば規約が守られる、という状態にはならない。
- 区分はこのページにだけ置く。各規約ページは規約そのものを定め、区分は持たない。

## 区分の定義

| 区分 | 違反したときに起きること | 担保の主体 |
| --- | --- | --- |
| **強制** | 操作が拒否される。または、規約に反する状態がそもそも発生しない | Repository Rulesets / Required status checks / リポジトリ設定 / GitHub Environments / ワークフローの検証ステップ |
| **検知のみ** | 操作は通る。記録が残るか、警告として可視化される | 監査ログ / 非ブロッキングの CI / 依存更新ボット / GitHub Deployments の履歴 |
| **運用** | 何も起きない。レビューで気付かなければそのまま通る | 人（レビュアー・リリース責任者） |

> [!WARNING]
> **「検知のみ」は担保ではなく材料である**
>
> 記録も警告も、見る人がいてはじめて意味を持つ。誰も見ない検知は「運用」と同じ強度しかない。段階導入で非ブロッキングにした検査を、そのまま放置しないこと（[依存とサプライチェーン](./supply-chain)）。

## 強制される規約

| 規約 | 担保手段 | 出典 |
| --- | --- | --- |
| `main` / `release/*` への直接 push を認めず、変更を PR 経由に限る | Ruleset: Restrict direct pushes | [ブランチ保護](./branch-protection) |
| PR に 1 名以上の承認を要する（作成者は自分の PR を承認できない） | Ruleset: Required approvals | [ブランチ保護](./branch-protection) |
| 承認後に push されたら、それまでの承認を無効化する | Ruleset: Dismiss stale approvals on push | [ブランチ保護](./branch-protection) |
| CI の成功をマージ条件とする | Ruleset: Required status checks | [ブランチ保護](./branch-protection) |
| レビューコメントを未解決のまま残してマージしない | Ruleset: Require conversation resolution | [ブランチ保護](./branch-protection) |
| マージコミットを積まない | Ruleset: Require linear history | [ブランチ保護](./branch-protection) |
| 履歴の書き換えとブランチの削除を認めない | Ruleset: Block force pushes / deletions | [ブランチ保護](./branch-protection) |
| 影響の大きいパスの変更にドメイン責任者の承認を要する | Ruleset: CODEOWNERS review | [ブランチ保護](./branch-protection) |
| `release/*` を作れる主体をリリース責任者に限る | Ruleset: creation restriction | [ブランチ保護](./branch-protection) |
| `v*` タグを作れる主体をリリース責任者に限る | Ruleset: タグの許可アクター | [ブランチ保護](./branch-protection) |
| 公開済みタグの更新・削除を誰にも認めない | Ruleset: タグの更新・削除の禁止（bypass 対象なし） | [ブランチ保護](./branch-protection) |
| 更新ボットが作った PR をレビューなしでマージしない | Ruleset: Required approvals（ボットの PR も同じ経路を通る） | [依存とサプライチェーン](./supply-chain) |
| PR タイトルを Conventional Commits に準拠させる | PR タイトルを検証する CI を Required status checks に含める | [マージルール](./merge-rules) |
| 検証を通ったタイトルをそのまま `main` のコミットメッセージにする | リポジトリ設定（squash のコミットメッセージを PR のタイトルと本文に固定） | [マージルール](./merge-rules) |
| `feature/*` → `main` を squash merge に固定する | リポジトリ設定（merge commit / rebase merge の無効化）+ Require linear history | [マージルール](./merge-rules) |
| RC タグの push で Pre-release と成果物を作る | タグ push を起点とするリリースワークフロー | [リリースとデプロイ](./release) |
| GA では再ビルドせず、RC 成果物を昇格させる | GA ワークフローが同一コミットとダイジェストを検証し、不一致なら失敗する | [リリースとデプロイ](./release) |
| `staging` / `production` へはタグ起点でのみデプロイする | Environments の deployment branch/tag policy | [リリースとデプロイ](./release) |
| `production` に RC を入れない | ワークフロー冒頭でタグ名が `-rc` を含む場合に失敗させる（タグのパターンに否定を書けないため） | [リリースとデプロイ](./release) |
| `production` へのデプロイに承認を要する（自己承認の禁止・管理者 bypass の無効化） | Environments の required reviewers / prevent self-review / wait timer | [リリースとデプロイ](./release) |
| `production` へのデプロイを直列化する | ワークフローの `concurrency` グループ | [リリースとデプロイ](./release) |
| 本番資格情報を、保護ルールを通過していないジョブから参照させない | Environment secrets のスコープ | [リリースとデプロイ](./release) |
| 本番ロールを `production` 以外のジョブに引き受けさせない | OIDC の trust policy が `sub` クレームに環境を要求する（クラウド側で強制） | [リリースとデプロイ](./release) |
| CI は lockfile 固定でインストールする | CI のインストールコマンド | [依存とサプライチェーン](./supply-chain) |
| 公開直後のバージョンを取り込まない | 依存更新ボットの cooldown 設定（推移的依存には効かない） | [依存とサプライチェーン](./supply-chain) |
| インストール時のライフサイクルスクリプトを実行しない | パッケージマネージャの設定（`ignore-scripts` 等） | [依存とサプライチェーン](./supply-chain) |
| しきい値以上の脆弱性を持つ依存を通さない | 差分検査と lockfile 全体検査の 2 つの CI ゲート | [依存とサプライチェーン](./supply-chain) |
| 2 つの検査のしきい値をそろえる | 値の一致を検査する CI | [依存とサプライチェーン](./supply-chain) |
| 脆弱性の例外に理由と再評価期限を要求し、期限切れを失敗させる | 例外の設定を検査する CI | [依存とサプライチェーン](./supply-chain) |
| ワークフロー定義の静的解析を通す | actionlint 等（blocking） | [依存とサプライチェーン](./supply-chain) |
| 全下流リポジトリの互換性テスト合格を GA の条件に含める | RC 公開を起点に走る下流 CI と、GA 判定への組み込み | [顧客別カスタマイズ 🚧](./customization) |

## 検知のみの規約

| 規約 | 検知の手段と限界 | 出典 |
| --- | --- | --- |
| bypass 権限を恒常的に付与しない | 発生は監査ログに残り、事後レビューの対象になる。bypass の実行そのものは止まらない | [ブランチ保護](./branch-protection) |
| Actions をコミット SHA でピン留めする | Actions のセキュリティ監査（zizmor 等）が検知する。段階導入の期間は非ブロッキング | [依存とサプライチェーン](./supply-chain) |
| ワークフローの権限を最小化する（`persist-credentials: false` と `timeout-minutes` を含む） | 同上 | [依存とサプライチェーン](./supply-chain) |
| `pull_request_target` で PR head をチェックアウト・実行しない | 同上 | [依存とサプライチェーン](./supply-chain) |
| PR タイトル・本文・ブランチ名を `run:` へ直接展開しない | 同上 | [依存とサプライチェーン](./supply-chain) |
| サポート中の release へ cherry-pick する | `needs-backport/vX.Y` ラベルを起点にバックポート追跡 Issue が自動生成される。追跡するだけで、cherry-pick 自体は強制されない | [障害対応](./incident) |
| `production` への承認とロールバックを記録する | GitHub Deployments の履歴に残る。履歴を正式な承認記録として扱う運用とセットで意味を持つ | [リリースとデプロイ](./release) |

## 運用に委ねる規約

機械的に担保できない理由は 2 つに大別できる。**GitHub 側に該当する機能が無い**か、**判断が要る**かである。前者は検査を書けば強制へ移せる余地があり、後者は移せない。

| 規約 | 機械的に担保できない理由 | 出典 |
| --- | --- | --- |
| 修正はまず `main` に入れる（upstream first） | 変更の由来を GitHub は判定しない。`release/*` 側で書いた修正かどうかは差分を読んで判断する | [ブランチ運用](./branching) |
| `main` → `release/*` は cherry-pick のみとし、merge / rebase で取り込まない | release ブランチへの PR は正規の経路であり、その中身が cherry-pick か merge かを区別する設定は無い | [マージルール](./merge-rules) |
| `release/*` → `main` のマージを禁止する | 同上。PR の向きそのものは正規の操作として通る | [マージルール](./merge-rules) |
| cherry-pick PR に元 PR へのリンクを書く | PR テンプレートは項目を並べるだけで、記入の有無を検査しない | [ブランチ保護](./branch-protection) |
| `main` に該当コードが無い場合の直接 PR に、その理由と再発しない根拠を書く | 同上。根拠が妥当かは読まないと分からない | [ブランチ保護](./branch-protection) |
| backport 先を保守中かつ該当バグのある release に絞る | どの release にバグが存在するかは判断が要る | [ブランチ運用](./branching) |
| PR タイトルの要約を利用者視点で具体的に書く | CI が検査するのは書式であり、`修正` のような内容を持たない要約も書式としては通る | [マージルール](./merge-rules) |
| ブランチ内の個々のコミットも Conventional Commits に揃える（推奨） | squash により `main` へ残らないため、検証の対象にしていない | [マージルール](./merge-rules) |
| long-lived な feature ブランチを作らない | ブランチの寿命に上限を課す設定は無い | [マージルール](./merge-rules) |
| PR を小さく保ち、大きくなるなら分割する | 変更行数の上限をマージ条件にはしていない | [マージルール](./merge-rules) |
| ブランチ名を `feature/<issue番号>-<短い説明>` 等に揃える | 命名を強制するルールは置いていない | [ブランチ運用](./branching) |
| PR に種別ラベルを付ける | ラベルの付与をマージ条件にする設定は無い。自動付与を用意しても、対応表に無い type は付かない | [リリースとデプロイ](./release) |
| 手動で GitHub Release を作らない | GitHub の画面から Release を作る経路は塞げない | [リリースとデプロイ](./release) |
| 自動生成したリリースノートを公開前に確定する | 判断が要る | [リリースとデプロイ](./release) |
| `-rc.N` サフィックスを成果物へ焼き込まない | 成果物の中身を検査する仕組みは置いていない | [リリースとデプロイ](./release) |
| 未完成機能を隔離手段のいずれかで到達不能にし、PR 本文に記載する | 到達不能かどうかはコードを読まないと分からない。`release/vX.Y` を切るリリース責任者が確認する | [バージョン運用 🚧](./versioning) |
| SemVer に従ってバージョンを決める | 変更が MAJOR / MINOR / PATCH のどれにあたるかは判断が要る | [バージョン運用 🚧](./versioning) |
| サポート対象を N / N-1 に保ち、EOL の release をアーカイブする | 判断と時期の管理が要る | [バージョン運用 🚧](./versioning) |
| セルフホストのアップグレードで MINOR を飛ばさない | 顧客環境での操作であり、リポジトリ側から強制できない | [バージョン運用 🚧](./versioning) |
| マイグレーションを expand-contract（後方互換）で書く | 後方互換かどうかを機械判定する仕組みは置いていない | [障害対応](./incident) |
| ロールフォワードを第 1 選択とし、ロールバックの可否を判断する | 障害の性質に依存する判断 | [障害対応](./incident) |
| 本番環境へ out-of-band の操作をしない | クラウド側のコンソール権限の問題であり、GitHub からは見えない | [障害対応](./incident) |
| 依存追加 PR に課題・検討した代替・保守状況を書く | 記入の有無も内容の妥当性も検査していない | [依存とサプライチェーン](./supply-chain) |
| 依存を増やさない選択肢を先に検討する | 判断が要る | [依存とサプライチェーン](./supply-chain) |
| 本番資格情報をリポジトリシークレットに格納しない | 格納すること自体は可能。参照範囲は環境で絞れるが、置き場所の選択は人が行う | [依存とサプライチェーン](./supply-chain) |
| 脆弱性の例外の理由に「なぜ影響が無いか」を書く | CI が検査するのは理由の**存在**であり、中身の妥当性ではない | [依存とサプライチェーン](./supply-chain) |
| 顧客識別子によるコード分岐を書かない | レビューで見る | [顧客別カスタマイズ 🚧](./customization) |
| Tier を判定し、Tier 2 に承認を要する | Tier の判定は設計上の判断である。承認そのものは CODEOWNERS で近似できる | [顧客別カスタマイズ 🚧](./customization) |
| 下流の修正遅延で GA が止まる場合の除外を判断し、記録する | 判断が要る | [顧客別カスタマイズ 🚧](./customization) |

> [!NOTE]
> **検討中のページに由来する項目**
>
> [バージョン運用](./versioning) と [顧客別カスタマイズ](./customization) は検討中（🚧）であり、規約が確定していない。両ページを出典とする項目は、区分も暫定である。

## 運用に委ねた規約をどう扱うか

「運用」の項目は、規約に書いただけでは守られない。次の 3 つで扱う。

1. **レビュー観点として明文化する。** 上の表の「運用」列は、そのままレビュー時に見る項目になる。PR テンプレートのチェックリストへ落とすと、書き手と読み手の双方に同じ観点が出る。
2. **確認の時点を決める。** 未完成機能の隔離やサポート対象の見直しのように、PR 単位では判断できない項目がある。これらは `release/vX.Y` を切る時点と GA 判定の時点に集約し、リリース責任者の確認事項として扱う。
3. **移せるものは強制へ移す。** 理由が「GitHub 側に機能が無い」であれば、ワークフローで検査を書けば強制へ移せる。理由が「判断が要る」であれば移せないため、人が見ることを前提に運用を組む。この区別が、上の表の 2 列目を書き分けている目的である。
