# 規約の担保状況

本セクションの各規約が、何によって守られるかを規約ページごとに一覧にする。

> [!NOTE]
> **一覧に書くのは「この規約が想定する担保手段」である**
>
> 本セクションは架空の製品の開発規約であり、このチュートリアルのリポジトリに現在入っている設定を説明したものではない。以下の表も、規約が前提とする設定・ワークフローを並べたものである。自分のリポジトリへ持ち込むときは、対応する設定が実在するかを別途確認すること。

## このページの要点

- 規約は担保のされ方で 3 つに分かれる。**強制**（違反すると止まる）・**検知のみ**（止まらないが記録される）・**運用**（人が守る）。
- 内訳は強制 31 件・検知のみ 7 件・運用 29 件。項目数では**運用**が多く、設定を入れれば規約が守られる、という状態にはならない。
- 区分はこのページにだけ置く。各規約ページは規約そのものを定め、区分は持たない。

## 区分の定義

| 区分 | 違反したときに起きること | 担保の主体 |
| --- | --- | --- |
| **強制** | 操作が拒否される。または、規約に反する状態がそもそも発生しない | Repository Rulesets / Required status checks / リポジトリ設定 / GitHub Environments / ワークフローの検証ステップ |
| **検知のみ** | 操作は通る。記録が残るか、警告として可視化される | 監査ログ / 非ブロッキングの CI / 依存更新ボット / GitHub Deployments の履歴 |
| **運用** | 何も起きない。レビューで気付かなければそのまま通る | 人（レビュアー・リリース責任者） |

> [!WARNING]
> **「検知のみ」は担保ではなく材料である**
>
> 記録も警告も、見る人がいてはじめて意味を持つ。誰も見ない検知は「運用」と同じ強度しかない。段階導入で非ブロッキングにした検査を、そのまま放置しないこと（[依存とサプライチェーン](./supply-chain)）。

以降、規約ページごとに整理する。3 列目は、区分が **強制** / **検知のみ** なら担保手段を、**運用** ならなぜ機械的に担保できないかを書く。

## [ブランチ運用](./branching)

| 規約 | 区分 | 担保手段 / 担保できない理由 |
| --- | --- | --- |
| 修正はまず `main` に入れる（upstream first） | 運用 | 変更の由来を GitHub は判定しない。`release/*` 側で書いた修正かどうかは差分を読んで判断する |
| backport 先を、保守中かつ該当バグのある release に絞る | 運用 | どの release にバグが存在するかは判断が要る |
| ブランチ名を `feature/<issue番号>-<短い説明>` 等に揃える | 運用 | 命名を強制するルールは置いていない |

## [ブランチ保護](./branch-protection)

| 規約 | 区分 | 担保手段 / 担保できない理由 |
| --- | --- | --- |
| `main` / `release/*` への直接 push を認めず、変更を PR 経由に限る | 強制 | Ruleset: Restrict direct pushes |
| PR に 1 名以上の承認を要する（作成者は自分の PR を承認できない） | 強制 | Ruleset: Required approvals |
| 承認後に push されたら、それまでの承認を無効化する | 強制 | Ruleset: Dismiss stale approvals on push |
| CI の成功をマージ条件とする | 強制 | Ruleset: Required status checks |
| レビューコメントを未解決のまま残してマージしない | 強制 | Ruleset: Require conversation resolution |
| マージコミットを積まない | 強制 | Ruleset: Require linear history |
| 履歴の書き換えとブランチの削除を認めない | 強制 | Ruleset: Block force pushes / deletions |
| 影響の大きいパスの変更にドメイン責任者の承認を要する | 強制 | Ruleset: CODEOWNERS review |
| `release/*` を作れる主体をリリース責任者に限る | 強制 | Ruleset: creation restriction |
| `v*` タグを作れる主体をリリース責任者に限る | 強制 | Ruleset: タグの許可アクター |
| 公開済みタグの更新・削除を誰にも認めない | 強制 | Ruleset: タグの更新・削除の禁止（bypass 対象なし） |
| bypass 権限を恒常的に付与しない | 検知のみ | 発生は監査ログに残り、事後レビューの対象になる。bypass の実行そのものは止まらない |
| cherry-pick PR に元 PR へのリンクを書く | 運用 | PR テンプレートは項目を並べるだけで、記入の有無を検査しない |
| `main` に該当コードが無い場合の直接 PR に、その理由と再発しない根拠を書く | 運用 | 同上。根拠が妥当かは読まないと分からない |

## [マージルールと PR タイトル規約](./merge-rules)

| 規約 | 区分 | 担保手段 / 担保できない理由 |
| --- | --- | --- |
| `feature/*` → `main` を squash merge に固定する | 強制 | リポジトリ設定（merge commit / rebase merge の無効化）+ Require linear history |
| PR タイトルを Conventional Commits に準拠させる | 強制 | PR タイトルを検証する CI を Required status checks に含める |
| 検証を通ったタイトルをそのまま `main` のコミットメッセージにする | 強制 | リポジトリ設定（squash のコミットメッセージを PR のタイトルと本文に固定） |
| `main` → `release/*` は cherry-pick のみとし、merge / rebase で取り込まない | 運用 | release ブランチへの PR は正規の経路であり、その中身が cherry-pick か merge かを区別する設定は無い |
| `release/*` → `main` のマージを禁止する | 運用 | 同上。PR の向きそのものは正規の操作として通る |
| PR タイトルの要約を利用者視点で具体的に書く | 運用 | CI が検査するのは書式であり、`修正` のような内容を持たない要約も書式としては通る |
| ブランチ内の個々のコミットも Conventional Commits に揃える（推奨） | 運用 | squash により `main` へ残らないため、検証の対象にしていない |
| long-lived な feature ブランチを作らない | 運用 | ブランチの寿命に上限を課す設定は無い |
| PR を小さく保ち、大きくなるなら分割する | 運用 | 変更行数の上限をマージ条件にはしていない |

## [バージョン運用 🚧](./versioning)

| 規約 | 区分 | 担保手段 / 担保できない理由 |
| --- | --- | --- |
| 未完成機能を隔離手段のいずれかで到達不能にし、PR 本文に記載する | 運用 | 到達不能かどうかはコードを読まないと分からない。`release/vX.Y` を切るリリース責任者が確認する |
| SemVer に従ってバージョンを決める | 運用 | 変更が MAJOR / MINOR / PATCH のどれにあたるかは判断が要る |
| サポート対象を N / N-1 に保ち、EOL の release をアーカイブする | 運用 | 判断と時期の管理が要る |
| セルフホストのアップグレードで MINOR を飛ばさない | 運用 | 顧客環境での操作であり、リポジトリ側から強制できない |

## [リリースとデプロイ](./release)

| 規約 | 区分 | 担保手段 / 担保できない理由 |
| --- | --- | --- |
| RC タグの push で Pre-release と成果物を作る | 強制 | タグ push を起点とするリリースワークフロー |
| GA では再ビルドせず、RC 成果物を昇格させる | 強制 | GA ワークフローが同一コミットとダイジェストを検証し、不一致なら失敗する |
| `staging` / `production` へはタグ起点でのみデプロイする | 強制 | Environments の deployment branch/tag policy |
| `production` に RC を入れない | 強制 | ワークフロー冒頭でタグ名が `-rc` を含む場合に失敗させる（タグのパターンに否定を書けないため） |
| `production` へのデプロイに承認を要する（自己承認の禁止・管理者 bypass の無効化） | 強制 | Environments の required reviewers / prevent self-review / wait timer |
| `production` へのデプロイを直列化する | 強制 | ワークフローの `concurrency` グループ |
| 本番資格情報を、保護ルールを通過していないジョブから参照させない | 強制 | Environment secrets のスコープ |
| 本番ロールを `production` 以外のジョブに引き受けさせない | 強制 | OIDC の trust policy が `sub` クレームに環境を要求する（クラウド側で強制） |
| `production` への承認とロールバックを記録する | 検知のみ | GitHub Deployments の履歴に残る。履歴を正式な承認記録として扱う運用とセットで意味を持つ |
| PR に種別ラベルを付ける | 運用 | ラベルの付与をマージ条件にする設定は無い。自動付与を用意しても、対応表に無い type は付かない |
| 手動で GitHub Release を作らない | 運用 | GitHub の画面から Release を作る経路は塞げない |
| 自動生成したリリースノートを公開前に確定する | 運用 | 判断が要る |
| `-rc.N` サフィックスを成果物へ焼き込まない | 運用 | 成果物の中身を検査する仕組みは置いていない |

## [障害対応](./incident)

| 規約 | 区分 | 担保手段 / 担保できない理由 |
| --- | --- | --- |
| サポート中の release へ cherry-pick する | 検知のみ | `needs-backport/vX.Y` ラベルを起点にバックポート追跡 Issue が自動生成される。追跡するだけで、cherry-pick 自体は強制されない |
| マイグレーションを expand-contract（後方互換）で書く | 運用 | 後方互換かどうかを機械判定する仕組みは置いていない |
| ロールフォワードを第 1 選択とし、ロールバックの可否を判断する | 運用 | 障害の性質に依存する判断 |
| 本番環境へ out-of-band の操作をしない | 運用 | クラウド側のコンソール権限の問題であり、GitHub からは見えない |

## [依存とサプライチェーン](./supply-chain)

| 規約 | 区分 | 担保手段 / 担保できない理由 |
| --- | --- | --- |
| CI は lockfile 固定でインストールする | 強制 | CI のインストールコマンド |
| 公開直後のバージョンを取り込まない | 強制 | 依存更新ボットの cooldown 設定（推移的依存には効かない） |
| インストール時のライフサイクルスクリプトを実行しない | 強制 | パッケージマネージャの設定（`ignore-scripts` 等） |
| しきい値以上の脆弱性を持つ依存を通さない | 強制 | 差分検査と lockfile 全体検査の 2 つの CI ゲート |
| 2 つの検査のしきい値をそろえる | 強制 | 値の一致を検査する CI |
| 脆弱性の例外に理由と再評価期限を要求し、期限切れを失敗させる | 強制 | 例外の設定を検査する CI |
| 更新ボットが作った PR をレビューなしでマージしない | 強制 | Ruleset: Required approvals（ボットの PR も同じ経路を通る） |
| ワークフロー定義の静的解析を通す | 強制 | actionlint 等（blocking） |
| Actions をコミット SHA でピン留めする | 検知のみ | Actions のセキュリティ監査（zizmor 等）が検知する。段階導入の期間は非ブロッキング |
| ワークフローの権限を最小化する（`persist-credentials: false` と `timeout-minutes` を含む） | 検知のみ | 同上 |
| `pull_request_target` で PR head をチェックアウト・実行しない | 検知のみ | 同上 |
| PR タイトル・本文・ブランチ名を `run:` へ直接展開しない | 検知のみ | 同上 |
| 依存追加 PR に課題・検討した代替・保守状況を書く | 運用 | 記入の有無も内容の妥当性も検査していない |
| 依存を増やさない選択肢を先に検討する | 運用 | 判断が要る |
| 本番資格情報をリポジトリシークレットに格納しない | 運用 | 格納すること自体は可能。参照範囲は環境で絞れるが、置き場所の選択は人が行う |
| 脆弱性の例外の理由に「なぜ影響が無いか」を書く | 運用 | CI が検査するのは理由の**存在**であり、中身の妥当性ではない |

## [顧客別カスタマイズ 🚧](./customization)

| 規約 | 区分 | 担保手段 / 担保できない理由 |
| --- | --- | --- |
| 全下流リポジトリの互換性テスト合格を GA の条件に含める | 強制 | RC 公開を起点に走る下流 CI と、GA 判定への組み込み |
| 顧客識別子によるコード分岐を書かない | 運用 | レビューで見る |
| Tier を判定し、Tier 2 に承認を要する | 運用 | Tier の判定は設計上の判断である。承認そのものは CODEOWNERS で近似できる |
| 下流の修正遅延で GA が止まる場合の除外を判断し、記録する | 運用 | 判断が要る |

## 運用に委ねた規約をどう扱うか

「運用」の項目は、規約に書いただけでは守られない。次の 3 つで扱う。

1. **レビュー観点として明文化する。** 各表の「運用」の行は、そのままレビュー時に見る項目になる。PR テンプレートのチェックリストへ落とすと、書き手と読み手の双方に同じ観点が出る。
2. **確認の時点を決める。** 未完成機能の隔離やサポート対象の見直しのように、PR 単位では判断できない項目がある。これらは `release/vX.Y` を切る時点と GA 判定の時点に集約し、リリース責任者の確認事項として扱う。
3. **移せるものは強制へ移す。** 理由が「GitHub 側に機能が無い」であれば、ワークフローで検査を書けば強制へ移せる。理由が「判断が要る」であれば移せないため、人が見ることを前提に運用を組む。この区別が、各表の 3 列目を書き分けている目的である。
